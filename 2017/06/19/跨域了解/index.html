跨域了解

发表于 2017-06-19

前言

浏览器同源策略的限制下，从一个域名的网页请求另一域名的资源：比如从www.baidu.com 页面去请求www.google.com的资源。这就所谓的跨域。

跨域问题

什么是跨域

概念：只要是协议、域名、端口中有任何一个不同都被当做是不同的域。

url 说明 是否允许通信
http://www.a.com/a.js
http://www.a.com/b.js 同一域名下 允许
http://www.a.com/lab/a.js
http://www.a.com/script/b.js 同一域名下不同文件夹 允许
http://www.a.com:8000/a.js
http://www.a.com/b.js 同一域名，不同端口 不允许
http://www.a.com/a.js
https://www.a.com/b.js 同一域名，不同协议 不允许
http://www.a.com.a.js
http://70.32.9.108/b.js 域名和域名对应IP 不允许
http://www.a.com/a.js
http://script.a.com/b.js 主域名相同，子域不同 不允许
http://www.a.com/a.js
http:a.com/b.js 同一域名，不同二级域名（同上） 不允许
http://www.cnblohs.com/a.js
http://www.a/com/b.js 不同域名 不允许

在协议和端口不一样的情况下，只能通过后台来解决。

为什么限制跨域

原因在于安全问题，比如：

  • 用户访问www.xxxxxx.com ，登录进行网银操作，这时cookie等生成并存放在浏览器;
  • 用户迷迷糊糊得访问了某个恶意网站www.eyi.com;
  • 这时这个恶意网站可以拿到银行的cookie，比如用户名、登录token等，然后发起对www.xxxxx.com操作；
  • 这时的浏览器和银行后台没做响应安全处理，用户的信息可能就存在泄漏的危险。

为什么跨域

既然有危险，那平常的项目中为什么要发起跨域请求呢？这时因为公司内部有多个不同的子域，比如
location.company.com ,而应用是放在app.company.com，这时后者去访问前者的资源就属于跨域了。

跨域通信访问

跨域通信访问的手段大概有：

  • jsonp （但向）
  • document.domain
  • window.name
  • hash传值(两个iframe之间)
  • possMessage
  • cors (Access-Control-Allow-Origin) (跨域资源共享)

jsonp

  • 关键点：浏览器对XMLHttprequest做了同源策策略处理，但没有把这种方式延续到script上（还有iframe,img,link等），从而可以利用动态script
    标签来做跨域请求。
    jsonp 由两部分组成 ：回调函数和数据，回调函数是当响应到来时应该在页面中调用的函数，而数据就是传入函数中的json数据。
  • 兼容性：所有浏览器兼容这种方法
  • 优点：前端可以很轻松的做到跨域请求
  • 缺点：
    1：只能通过GET方式请求，参数长度有限制，安全性比较差；
2：后端需要知道前端的cd是什么样的结构，主要是参数和回调名；
3：后端需要进行参数和cd的拼接然后才能执行。
  • 使用方式：

document.domain

  • 关键点：跨域分两种，一种是xhr不能访问不同源的文档，另一种是不同域的框架之间是不能进行JS交互的。但不同框架之间是可以获取到window对象的，但无法获取响应的属性和方法。
    这个时候可以利用document.domain解决这种情况，但且只能适用于主域相同，子域不同的情况，
    且只能把docment.domain设置成自身或更高一级的父域，且主域必须相同。例如 a.b.example.com中某一个文档的document.domain可以设置为a.b.example.com、b.example.com、
    example.com中的任意一个，但不能设置为c.a.b.example.com 因为这是当前域的子域，也不可以设置为baidu.com,因为主域已经不同。

  • 兼容性：所有浏览器都支持；

  • 优点：可以实现不同window之间的相互访问和操作
  • 缺点：只使用于父子window之间，不能用于xhr，只能在主域相同且子域不同的情况下用。
  • 使用方法：
    1. a(当前页面或父页面) 页面中加入 document.domain = ‘example.com’;
    2. b(当前页面或子页面) 页面中加入 document.domain = ‘example.com’;
    3. a页面访问b页面里面的数据或者方法。

window.name

  • 关键点：window对象的name属性，在一个窗口(window)的生命周期内，窗口载入的所有页面共享这个window.name，每个页面对其都有读写的权限。
  • 兼容性：所有浏览器都支持
  • 优点：
    1. 最简单的利用了浏览器的特性来做到不同域之间的数据传递；
    2. 不需要前后端的特殊匹配设置。
  • 缺点：
    1. 大小限制：window.name最大size为2M左右，不同的浏览器有不同的约定；
    2. 安全性：当前页面所有window都可以作修改，很不安全；
    3. 数据类型：传递数据只限于字符串，如果是对象或者其他会被自动转化为字符串
  • 使用方式：直接修改window.name即可。

hash

  • 关键点：URL的#号及其后的字符这一部分称为hash，它一般用于锚点定位，server端不关心这部分，因为HTTP请求过程不会携带上hash，但这部分还是会产生浏览器历史记录。
  • 兼容性： 有些浏览器不支持onhashchange事件，需要轮询来获取知URL的改变。
  • 缺点：数据直接暴露在URL中，数据容量和类型有限。
  • 使用方式：
    假如父页面是baidu.com/a.html,iframe嵌入的页面为google.com/b.html(此次省略了域名url属性)，要实现此两个页面间的通信可通过下面的方法：
    1. a.html传送数据到b.html
      (1) a.html下修改iframe的src为google.com/b.html#paco;
      (2) b.html监听到url发生变化，触发相应的操作
    2. b.html传送数据到a.html，由于两个页面不在同一个域下，(IE和Chrome不允许修改paren.location.hash的值，所有需要借助父窗口域名下的一个代理iframe)
      (1) b.html下创建一个隐藏的ifram，此ifram的src是baidu.com域下的，并挂上要传送的hash数据如：src=’http://www.baidu.com/peoxy.html#data‘
      (2) proxy.html监听到url的变化，修改a.html的url(因为a.html和proxy.html同域，所以proxy.html可修改a.html的url hash)
      (3) a.html 监听到url发生变化，触发相应的操作

possMessage

  • 关键点：possMessage是HTML5新引入的特性，可以使用它来向其他的window对象发送消息，无论这个对象是否在同源或不同源，
  • 兼容性：移动端可放心使用，PC端IE8以下不支持
  • 优点：不需要后端借入，即可简单的进行跨域；移动端兼容性好。
  • 缺点：
    1. 无法做到一对一的传递方式，监听中需要做消息识别，同一页面中的消息，onmessage都会收到；
    2. 安全性问题：第三方可以通过截获，注入html或者脚本的形式监听到消息，从而做到篡改的效果，所有要做好限制功能；
    3. 发送的数据有要求，如function就不能当做参数进行传递
  • 使用方式：
    otherWindow.postMessage(message,targetOrigin);
    otherWindow: 目标窗口，即就是给哪个windo发送消息，window.frames属性的成员或者window.open方法创建的窗口;
    message:是要发送的消息，类型为String,Objext(IE8，9不支持)；
    targetOrigin:是限定接收范围，不限制额使用’*’;
    B页面通过message事件监听并接收消息。

cors

  • 关键点：cors是一种通过前后端自定义http header配置来进行跨域的一种方式，
  • 兼容性：移动端兼容良好，PC端兼容IE11；
  • 优点：前端方便，只需发请求，不用考虑跨域问；安全性得到控制
  • 缺点：兼容性不全面，需要做降级处理
  • 使用方式：
    服务器端对于使用cors的支持，主要是通过设置Access-Control-Allow-Origin来进行，如果浏览器检测到相应的设置，就可以允许Ajax进行跨域的访问。
    前端进行正常请求，无论是用xhr或者其他。
